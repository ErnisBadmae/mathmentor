# Simplicity First

## Главный принцип

Сложность накапливается. 100 строк превращаются в 1000, потом в 10000 — и код перестаёт поддаваться правке ни человеком, ни моделью.

**Каждое изменение должно уменьшать сложность или оставлять её прежней. Увеличение — только с явным обоснованием.**

## Что это значит на практике

**Перед тем как писать код — спроси себя:**

- Эта абстракция решает реальную проблему сейчас, или "на будущее"?
- Можно ли решить это существующим инструментом/паттерном без нового слоя?
- Если удалить этот код — что сломается? Если ничего — удали.

**Запрещено без явной просьбы пользователя:**

- Вводить новые абстракции, базовые классы, фабрики, registry-паттерны
- Добавлять параметры "для гибкости" которые сейчас не нужны
- Разбивать простую функцию на несколько "для чистоты"
- Добавлять слои индирекции (обёртки над обёртками)
- Писать обобщённый код вместо конкретного

**Признаки accidental complexity — исправлять при встрече:**

- Функция делает одно действие, но называется абстрактно (`process`, `handle`, `manage`)
- Класс существует только чтобы вызвать один метод
- Конфигурация есть, но значение никогда не меняется
- Комментарий объясняет КАК, а не ПОЧЕМУ

## Скрытие сложности

Если сложность неизбежна (протокол, внешний API, легаси) — **прячь её за узким интерфейсом**.
Снаружи должно быть просто. Сложность внутри — нормально. Сложность на поверхности — нет.

## Размер

- Функция > 40 строк — повод задуматься, не повод рефакторить автоматически
- Файл > 300 строк — смотри есть ли явные независимые части
- PR > 400 строк изменений — предложи разбить, если это не одна атомарная фича

## Применение

Это правило активно **всегда**: при добавлении фич, исправлении багов, рефакторинге.
Не нужно ждать явной просьбы "упрости код" — не усложняй по умолчанию.
